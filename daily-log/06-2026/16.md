## Learning

### Java Garbage Collection

Today I explored how Java manages memory automatically through Garbage Collection (GC).

#### Comparison with C/C++

In C and C++, memory allocated for objects must be manually released by the developer using mechanisms such as `free()` or `delete()`.

If memory is not released properly:

* Memory leaks can occur.
* The application may eventually run out of memory.

#### How Java Handles It

Java provides Automatic Garbage Collection.

* Objects are created on the Heap.
* When an object no longer has any reachable references, it becomes eligible for garbage collection.
* The JVM automatically identifies and removes such objects.
* This frees heap memory without requiring manual intervention from developers.

#### Key Takeaway

Garbage Collection does **not** immediately destroy an object when its reference becomes `null`.

Instead:

1. The object becomes eligible for garbage collection.
2. The JVM decides when to reclaim that memory.
3. The exact timing is not guaranteed.

#### Example

```java
Employee emp = new Employee();
emp = null;
```

After `emp = null`, the object becomes **eligible for garbage collection**, but it is **not guaranteed to be removed immediately**.
